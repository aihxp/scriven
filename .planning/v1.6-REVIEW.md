## HIGH

### H-01 — `atomicWriteFileSync` does not fsync the parent directory
**File:** `bin/install.js:486-506`

After `fs.renameSync(tmpPath, targetPath)`, the directory entry is only durable after the containing directory's metadata is flushed. On crash between rename and directory flush, some filesystems (notably ext4 with `data=writeback`, and some network filesystems) can lose the rename even though file contents are fsynced. The "atomic" claim is half-complete without parent-dir fsync.

**Fix:**
```js
fs.renameSync(tmpPath, targetPath);
try {
  const dfd = fs.openSync(dir, 'r');
  try { fs.fsyncSync(dfd); } finally { fs.closeSync(dfd); }
} catch { /* best effort — Windows rejects dir fsync */ }
```

### H-02 — `copyDirWithPreservation` race window between hash check and rename; partial-write risk on copy
**File:** `bin/install.js:843-876`

Flow: (1) `sha256File(destPath)` → (2) `sha256File(srcPath)` → (3) `renameSync(destPath, backupPath)` → (4) `copyFileSync(srcPath, destPath)`. Between 1 and 4, a concurrent modifier could race. More importantly, `copyFileSync` at step 4 is **not** atomic — a crash mid-copy leaves `destPath` truncated with no backup and no tmp sibling to clean up. Same issue at `bin/install.js:860` (fresh) and `:866` (replaced).

**Fix:** Route the final copy through `atomicWriteFileSync`:
```js
const srcBuf = fs.readFileSync(srcPath);
fs.renameSync(destPath, backupPath);
atomicWriteFileSync(destPath, srcBuf);
```

## MEDIUM

### M-01 — Symlink handling in `copyDirWithPreservation` / `sha256File` is unexamined
**File:** `bin/install.js:833-841, 848-874`

`fs.readdirSync(..., { withFileTypes: true })` follows symlinks. A symlinked template file gets its target hashed, then the symlink is renamed to backup, and the shipped template is copied to the former symlink path. Worst current outcome is a symlink-to-secret rename inside the user's own data dir — not an escape — but code should reject or explicitly handle non-regular-file dest entries.

**Fix:** `fs.lstatSync(destPath)` gate before hashing; skip or unlink+fresh-copy if not a regular file.

### M-02 — `cleanOrphanedTempFiles` is non-recursive but tmp files can be nested
**File:** `bin/install.js:508-521, 1373-1379`

`atomicWriteFileSync` writes `tmp.<uuid>` next to the target, which may be deeply nested (e.g. `skillsDir/<skillName>/SKILL.md`). But `cleanOrphanedTempFiles` only scans the top-level dir from `collectTargetDirsForSweep`. An interrupted Codex skill install can leave orphans under `~/.codex/skills/scr-help/*.tmp.<uuid>` that survive forever.

**Fix:** Either recurse in `cleanOrphanedTempFiles` (with a depth cap) or add every atomic-write destination dir to `collectTargetDirsForSweep`. Recursion is safer.

### M-03 — `readFrontmatterValue` treats YAML block-scalar indicators (`|`, `>`) as literal values
**File:** `bin/install.js:342-344`

If a command file ever uses `description: |\n  multi\n  line`, the parser returns literal `|`. Downstream Codex skill metadata would silently ship broken.

**Fix:** Detect `|`/`>` as first non-whitespace non-comment token and either warn + fall back to key-as-description, or consume indented continuation lines. For v1.6 scope, the warn+fallback path is sufficient; add a regression test.

### M-04 — `mergeSettings` skips migration/validation on the existing file
**File:** `bin/install.js:900-908, 1300-1312`

`writeSharedAssets` reads existing settings via `readJsonIfExists` (swallows parse errors, returns `null`) and merges straight in. Old-format, hand-edited, or schemaful-but-invalid files are silently merged — junk fields propagate across installs. `readSettings` correctly migrate-then-validates; the merge path doesn't.

**Fix:** Before merging, run `migrateSettings` + `validateSettings`. If invalid, back up to `settings.json.invalid.<timestamp>` and write fresh, or log a warning and drop to fresh settings.

## LOW

### L-01 — UUID regex in `cleanOrphanedTempFiles` is looser than producer output
**File:** `bin/install.js:510`

`/\.tmp\.[0-9a-f-]{36}$/i` accepts 36 chars of `[0-9a-f-]` in any arrangement — a user file named `foo.tmp.<36 dashes>` would be deleted.

**Fix:** Tighten to canonical UUID shape: `/\.tmp\.[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i`.

### L-02 — `readFrontmatterValues` silently drops duplicate keys
**File:** `bin/install.js:337`

First occurrence wins. Real YAML raises or does last-wins. Silent first-wins can mask edit bugs in command files.

**Fix:** Warn on duplicate keys, or document the choice with a test.

### L-03 — `stripInlineComment` uses `search(/\s#/)` which matches newline whitespace too
**File:** `bin/install.js:316`

Lines are pre-split so not a bug today. Future refactors feeding multi-line strings would silently truncate.

**Fix:** Use `search(/[ \t]#/)`.

### L-04 — `isScrivenInstalledCommandFile` reads every `.md` in target dir per run
**File:** `bin/install.js:621-625, 636-642`

O(N) I/O per install on every command file to check for marker. Scales with command count.

**Fix:** Not required for v1.6; prefer manifest-first with content-sniff fallback.

### L-05 — `shellQuote` assumes POSIX single-quote escaping
**File:** `bin/install.js:27-29`

Fine today (Perplexity macOS via `/bin/sh`). Would break on Windows Perplexity if ever added. Not in scope.

## INFO

### I-01 — Installer-owned vs. schema field drift risk
**File:** `bin/install.js:887-898`

`SETTINGS_SCHEMA` and `INSTALLER_OWNED_FIELDS` are two hand-maintained lists. A future contributor adding a field must remember to classify ownership.

**Suggestion:** Tag ownership on schema entries (`owned_by: 'installer' | 'user'`) and derive `INSTALLER_OWNED_FIELDS`.

### I-02 — `generateCodexSkill` escapes `"` but not `\` in description
**File:** `bin/install.js:420-422`

`replace(/"/g, '\\"')` is fine for current content. A literal backslash in a description would emit an unescaped backslash in a YAML double-quoted scalar.

**Fix:**
```js
const yamlDouble = (s) => s.replace(/\\/g, '\\\\').replace(/"/g, '\\"');
```
Apply to both `description` and `short-description`.
