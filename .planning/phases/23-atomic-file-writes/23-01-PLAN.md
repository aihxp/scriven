<objective>
Make every installer file write crash-safe. An interrupted install (SIGKILL, power loss, disk-full after open) must never leave a truncated or partially-written file at a target path. Orphaned temp files from prior interrupted runs must be cleaned up before new writes start.

Purpose: Preserve installer reliability (SAFE-01, SAFE-02). Users re-running the installer after a crash should see a clean, working install — not a half-written settings.json or truncated SKILL.md that silently breaks runtime commands.

Output: Two new exported helpers in bin/install.js (atomicWriteFileSync, cleanOrphanedTempFiles), replacement of all output-path fs.writeFileSync calls with atomicWriteFileSync, and a startup sweep of orphan temp files across all runtime target directories.
</objective>

2. Insert the two helpers immediately after `removePathIfExists` (around line 426) and before `insertMarkerComment`:

```js
function atomicWriteFileSync(targetPath, content) {
  const dir = path.dirname(targetPath);
  fs.mkdirSync(dir, { recursive: true });
  const tmpPath = `${targetPath}.tmp.${crypto.randomUUID()}`;
  try {
    fs.writeFileSync(tmpPath, content);
    fs.renameSync(tmpPath, targetPath);
  } catch (err) {
    try { fs.unlinkSync(tmpPath); } catch { /* best effort */ }
    throw err;
  }
}

function cleanOrphanedTempFiles(dir) {
  if (!fs.existsSync(dir)) return 0;
  const TMP_PATTERN = /\.tmp\.[0-9a-f-]{36}$/i;
  let removed = 0;
  for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
    if (!entry.isFile()) continue;
    if (!TMP_PATTERN.test(entry.name)) continue;
    try {
      fs.unlinkSync(path.join(dir, entry.name));
      removed++;
    } catch { /* best effort */ }
  }
  return removed;
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| installer process -> filesystem | Installer generates file content in memory and writes to user-owned directories (~/.claude, ~/.scriven, project-local .codex, etc.). A crash between fs.writeFileSync and the buffer flushing to disk can truncate the target. |
| prior-run temp files -> current-run installer | An interrupted earlier install may have left `*.tmp.*` siblings in target directories. These are attacker-irrelevant (same user, same dirs) but must not accumulate or be mistaken for legitimate installer output. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-01 | Tampering | atomicWriteFileSync target path | mitigate | Temp path uses `crypto.randomUUID()` (cryptographically-random, unguessable within user's own dir) and `fs.renameSync` is POSIX-atomic — a concurrent process cannot observe a half-written target. |
| T-23-02 | Denial of Service | Orphaned temp files accumulating over many crashed installs | mitigate | `cleanOrphanedTempFiles` runs at installer startup across every target directory for the selected runtimes, bounded by the UUID suffix regex so only installer-generated orphans match. |
| T-23-03 | Tampering | cleanOrphanedTempFiles regex over-matches and deletes user files | mitigate | Regex is anchored to `\.tmp\.[0-9a-f-]{36}$` (exact UUID v4/v7 length + hex/dash charset). A user-named `notes.tmp.draft` will NOT match. Tests enforce this negative case. |
| T-23-04 | Information Disclosure | Temp file content exposed before rename | accept | Temp file is written to the same directory as the target with the same permissions inheritance; no new exposure surface beyond the existing target. User-only dirs (~/.claude, ~/.scriven) already scope visibility. |
| T-23-05 | Denial of Service | Mid-write crash leaves target path locked or inaccessible | mitigate | Because writes happen to `*.tmp.*` and only rename touches the target, the target either retains its complete prior content OR is replaced wholesale with the complete new content — it is never in an inaccessible/locked intermediate state. |
</threat_model>
