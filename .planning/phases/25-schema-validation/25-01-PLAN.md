<objective>
Build the settings schema validation infrastructure inside `bin/install.js`: a hand-written `SETTINGS_SCHEMA` constant, a `validateSettings` checker, a `migrateSettings` upgrader, and a `readSettings` pipeline that chains raw-read → migrate → validate. This closes SCHEMA-01 (validation with clear errors) and SCHEMA-02 (migration runs before validation).

Purpose: Catch misconfigurations in `settings.json` loudly and early, with field-name-level error messages, while letting older-format settings upgrade cleanly instead of failing on newly added fields. Prevents the bootstrap deadlock called out in PITFALLS.md §8 where a validator rejects the very config that migration would have fixed.

Output: Four new helpers in `bin/install.js` (schema constant + three functions), all exported from `module.exports`, plus a full regression test file covering valid/invalid/migration paths. No wiring into `runInstall` yet — that is Phase 26's job.
</objective>

    SETTINGS_SCHEMA shape — an object (or array of field descriptors) where each field declares:
      - name (string)
      - type ('string' | 'boolean' | 'array-of-string')
      - required (boolean)
      - enum (optional array of allowed values, e.g. ['global','project'] for scope, ['interactive','non-interactive'] for install_mode)
      - allow_empty (optional boolean, true for `runtime` which may be empty string)

    SETTINGS_SCHEMA fields:
      - version (string, required)
      - runtime (string, required, allow_empty=true)
      - runtimes (array-of-string, required)
      - scope (string, required, enum=['global','project'])
      - developer_mode (boolean, required)
      - data_dir (string, required)
      - install_mode (string, required, enum=['interactive','non-interactive'])
      - installed_at (string, required)

    migrateSettings(raw) behavior:
      - If raw is null/undefined, return null (caller decides what that means)
      - If raw.runtimes is missing but raw.runtime is a non-empty string, set raw.runtimes = [raw.runtime]
      - If raw.runtimes is missing and raw.runtime is missing/empty, set raw.runtimes = []
      - If raw.scope is missing, default to 'global'
      - If raw.install_mode is missing, default to 'interactive'
      - Return the mutated/new object (do not mutate the caller's input — clone first with `{ ...raw }`)
      - Do NOT invent values for version, data_dir, or installed_at (validator will catch missing ones)

    validateSettings(settings) behavior:
      - Returns `{ valid: boolean, errors: string[] }`
      - If settings is not a plain object → `{ valid: false, errors: ['settings: expected object, received <typeof>'] }`
      - For each field in SETTINGS_SCHEMA:
        - If required and the key is absent → push `"<field>: required field is missing"`
        - If present, check type:
          - 'string': typeof value === 'string'; if allow_empty is false and value === '', flag as invalid
          - 'boolean': typeof value === 'boolean'
          - 'array-of-string': Array.isArray(value) && every element is a string
        - Type mismatch → push `"<field>: expected <expected-type>, received <actual-type>"` where actual-type is `typeof value` or 'array'/'null' as appropriate
        - If enum defined and value not in enum → push `"<field>: expected one of [a, b], received <value>"`
      - For each key in settings not in SETTINGS_SCHEMA → push `"<field>: unknown field (warning)"` but do NOT flip valid to false
      - valid = (no non-warning errors). Implement this by tracking hard errors separately from warning strings, or by checking whether every error string ends with '(warning)'.

    readSettings(dataDir) behavior:
      - settingsPath = path.join(dataDir, 'settings.json')
      - raw = readJsonIfExists(settingsPath)
      - If raw is null → throw `new Error('settings.json not found at <path>')`
      - migrated = migrateSettings(raw)
      - result = validateSettings(migrated)
      - If !result.valid → throw `new Error('Invalid settings: ' + result.errors.filter(non-warning).join('; '))`
      - Return migrated

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| disk → installer | `settings.json` read from user's data dir. Content is user-influenced (users can hand-edit). Malformed or adversarial JSON must not crash the installer or be silently accepted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Tampering | `settings.json` on disk | mitigate | `validateSettings` rejects type mismatches and enum violations with field-level messages; `readSettings` throws on invalid config so callers cannot silently consume a tampered object. |
| T-25-02 | Denial of Service | `readSettings` on malformed JSON | mitigate | `readJsonIfExists` already catches JSON parse errors and returns null; `readSettings` treats null as "not found" and throws a clear error — no unhandled exception propagates. |
| T-25-03 | Information Disclosure | Error messages containing user data | accept | Error messages include field names and received type (e.g. `typeof`), not raw values for string fields. Values only appear in enum mismatches where the value is a controlled enum key, not free-form user data. Low risk. |
| T-25-04 | Elevation of Privilege | Unknown-field injection via settings.json | mitigate | Open-schema policy: unknown fields produce warnings but are preserved. Validation does not grant them any behavioral effect — they are inert until an explicit consumer reads them. Phase 26 will extend this when settings preservation ships. |
</threat_model>
