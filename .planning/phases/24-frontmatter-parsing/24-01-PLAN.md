  <behavior>
    - Given content starting with `---\n...\n---\n<body>`, only lines inside the fenced block are considered
    - Given content with no opening `---` on the first non-empty line, `readFrontmatterValue` returns ''
    - For `description: "Step 1: Do this"`, returns `Step 1: Do this` (colons preserved) — PARSE-01
    - For `description: Scan: files` (unquoted, colon in value), returns `Scan: files` — PARSE-01
    - Body text `description: body line` appearing AFTER the closing `---` is NOT matched — PARSE-02
    - `argument-hint: [work-id, locale]` returns `[work-id, locale]` verbatim as a string — PARSE-03
    - Missing key returns '' (empty string)
    - Quoted values (single or double) have wrapping quotes stripped (preserve current behavior via stripWrappingQuotes)
    - Inline trailing `# comment` on an unquoted value is stripped; `#` inside a quoted value is preserved
    - Split on FIRST colon only using `line.indexOf(':')` so keys cannot contain `:` but values can
    - New helper `readFrontmatterValues(content)` returns `{ [key]: string }` for all pairs in the block; used by future callers and enables efficient test coverage
  </behavior>
  <action>
Refactor `readFrontmatterValue` in `bin/install.js` (currently lines 283-296) as follows:

1. Keep `stripWrappingQuotes` unchanged.

2. Add internal helper `extractFrontmatterBlock(content)`:
   - Split `content` on `/\r?\n/`.
   - If the first line (after ignoring a UTF-8 BOM) is not exactly `---`, return `null`.
   - Walk lines starting at index 1; the block ends at the next line that is exactly `---` or `...`.
   - If no closing fence is found, return `null` (malformed — treat as no frontmatter).
   - Return the array of lines strictly BETWEEN the opening and closing fences.

3. Add `readFrontmatterValues(content)` that returns a plain object:
   - Call `extractFrontmatterBlock`; if `null`, return `{}`.
   - For each line:
     - Skip empty lines and lines starting with `#` (YAML comments).
     - Find `const idx = line.indexOf(':');` — if `idx === -1`, skip (do not throw).
     - `const key = line.slice(0, idx).trim();`
     - `let value = line.slice(idx + 1);`
     - Strip a trailing unquoted inline comment: if `value` (after trim) does NOT start with `"` or `'`, remove everything from the first ` #` onward. If it starts with a quote, leave `#` intact inside the quoted span.
     - `value = stripWrappingQuotes(value);`
     - If `key` is non-empty and not already set (first occurrence wins), assign `result[key] = value`.
   - Note: Multiline YAML (`|`, `>`) and continuation indentation are NOT required by any shipped command file today (verify via `grep -l "^description: |" commands/`). If a value is exactly `|` or `>`, return it as-is (defensive — callers treat empty-string fallback). Do NOT attempt full YAML semantics — this is a pragmatic subset per D-01 (Claude's discretion, research/SUMMARY.md).

4. Rewrite `readFrontmatterValue(content, key)` to:
   ```js
   function readFrontmatterValue(content, key) {
     const values = readFrontmatterValues(content);
     return Object.prototype.hasOwnProperty.call(values, key) ? values[key] : '';
   }
   ```
   This preserves the exported signature and return type (always string).

5. Update `module.exports` at the bottom of `bin/install.js` to additionally export `readFrontmatterValues` (and keep the existing `readFrontmatterValue` export). Grep for the existing `module.exports` block to confirm format.

6. Do NOT change any call site. `collectCommandEntries` continues to call `readFrontmatterValue(content, 'description')` / `'argument-hint'` unchanged.

Zero new dependencies. Node.js built-ins only (string methods). No regex catastrophic-backtracking risk because we split on newlines then scan characters.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| command file → installer | Installer reads arbitrary `.md` files under `commands/scr/` and extracts metadata; malformed or hostile frontmatter must not crash or mis-parse |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Tampering | `readFrontmatterValue` (body lines masquerading as frontmatter) | mitigate | Scope parsing strictly to the fenced `---` block via `extractFrontmatterBlock`; body lines are never inspected (PARSE-02) |
| T-24-02 | Information Disclosure | Command description rendered into Codex/Claude manifests | mitigate | Values are returned verbatim as strings; no interpolation or shell-exec path touches them; quotes are stripped but never unescaped beyond literal wrapping quote removal |
| T-24-03 | Denial of Service | Pathological input (missing closing fence, giant line) | mitigate | No unbounded regex; line-based scanner with O(n) cost. Missing closing fence returns `{}` rather than hanging |
| T-24-04 | Repudiation | Silent truncation producing wrong manifest entries | mitigate | Regression tests (PARSE-01/02/03) fail loudly if parser drifts (QA-05 is covered separately in Phase 28 but these tests already exercise the surface) |
</threat_model>
