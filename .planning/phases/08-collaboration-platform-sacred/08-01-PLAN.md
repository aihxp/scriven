Create `commands/scr/track.md` as a single command file with subcommand routing (like export.md pattern). The command wraps git branch/merge/diff operations in writer-friendly language per D-01 and D-02.

**Frontmatter:**
```yaml
description: Manage revision tracks — create, switch, compare, merge, and propose changes for review.
argument-hint: "<create|list|switch|compare|merge|propose> [name] [options]"
```

**Subcommand: create <name>**
- Creates a new revision track with the given writer-friendly label (e.g., "Editor's suggestions", "Second draft attempt")
- Internally: generates a sanitized git branch name from the label (slugify: lowercase, replace spaces with hyphens, strip special chars)
- Stores label-to-branch mapping in `.manuscript/tracks.json`: `{ "tracks": [{ "label": "Editor's suggestions", "branch": "track/editors-suggestions", "created": "ISO date", "author": "writer" }] }`
- Runs `git checkout -b track/{slug}` from current position
- Confirms: "Created revision track 'Editor's suggestions'. You're now working on it."
- If `config.json` has `collaboration.tracks_enabled: false`, set it to `true` on first track creation

**Subcommand: list**
- Reads `.manuscript/tracks.json` and shows all tracks with:
  - Track name (writer-friendly label)
  - Status: active (current), ahead/behind canon, last checkpoint date
  - Uses git commands to determine commit count ahead/behind main
- Always shows "Canon manuscript" as the base track (main/master branch)
- Marks the currently active track with an indicator

**Subcommand: switch <name>**
- Looks up the writer-friendly label in `tracks.json` to find the git branch
- Runs `git checkout {branch}`
- Confirms: "Switched to revision track 'Editor's suggestions'."
- If switching to canon: `git checkout main` (or master)
- If unsaved changes exist, warn and suggest `/scr:save` first

**Subcommand: compare [a] [b]**
- If both names given: compares track A vs track B
- If one name given: compares that track vs canon
- If no names: compares current track vs canon
- Resolves writer-friendly labels to branch names via `tracks.json`
- Uses `git diff` between branches
- Displays in writer-friendly format reusing the pattern from `compare.md`:
  - Show passage-by-passage changes (not raw diff)
  - Label sections with "In [track name]:" vs "In [other track]:"
  - Highlight additions in context, removals marked clearly
  - Summary: "X passages changed, Y passages added, Z passages removed"

**Subcommand: merge <name>**
- Merges the named track into canon (current must be on canon, or auto-switch to canon first)
- Resolves label to branch via `tracks.json`
- Attempts `git merge {branch}`
- If clean merge: "Revision track 'Second draft' merged into canon manuscript. All changes accepted."
- If conflicts (per D-02): Show each conflicting passage side-by-side with three options:
  - "Keep mine" (canon version)
  - "Keep theirs" (track version)
  - "Keep both" (concatenate both versions with a scene break)
  - Walk through conflicts one at a time, resolve each, then complete merge
- After merge: run continuity-check on merged content (suggest `/scr:continuity-check` if available)
- Update `tracks.json`: mark track as merged with merge date

**Subcommand: propose <name>**
- Creates a revision proposal from the named track for editor review
- Generates `.manuscript/proposals/{track-slug}-proposal.md` containing:
  - Track name, author, date
  - Summary of changes (auto-generated from diff: passages changed, added, removed)
  - Full diff in readable format
  - Space for editor notes (empty section for editor to fill)
- Confirms: "Revision proposal created for 'Editor's suggestions'. Share with your editor for review."

**Writer-friendly language throughout:**
- Never say "branch", "commit", "merge conflict", "checkout"
- Use: "revision track", "checkpoint", "continuity conflict", "switch to"
- Error messages in plain English: "You have unsaved changes. Save them first with /scr:save before switching tracks."
